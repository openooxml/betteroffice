# BetterOffice

## Comments

NO inline comments unless the code is not understandable without them. Docstrings must be extremely concise.

## PR style

PR bodies follow this template:

```
TL;DR:
<human (!) written tl;dr of the pr>

Before/After:
<only for visual changes: before and after screenshots>

Repro file:
<only for engine fixes/improvements: link to the file that reproduces the issue>

Summary:
<concise bullets, gen'd by agent>

Test plan:
<check-boxes the author must fulfill before pinging for review>
```

Section rules:

- **TL;DR** — written by a human, always. Agents must leave it empty and never fill it in.
- **Before/After** — include only when the PR makes visual changes. Attach a before screenshot and an after screenshot (drag them into the PR body so GitHub hosts them). Omit the section entirely otherwise.
- **Repro file** — include only when the PR fixes or improves something on the engine. Link the repro file (e.g. a committed test fixture or an uploaded `.docx`) that demonstrates the issue the PR addresses. Omit the section entirely otherwise.
- **Summary** — concise bullets describing what changed; generated by the agent. What changed, not how.
- **Test plan** — GitHub check-boxes (`- [ ]`). The author must fulfill every box before pinging for review.
